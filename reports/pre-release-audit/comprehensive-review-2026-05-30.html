1. Install / Chat / Update 機能

✅ 正常動作を確認した領域

  • 3 エージェント (ghcp, claude, codex) の検出 — detectAgents() が正しく動作
  • デフォルト (marketplace) プラグインインストール — runPluginOperation() で全エージェント対応
  • ローカルインストール (--local) — applySetup() によるファイルコピー方式
  • Chat 機能 — 全 3 エージェント向け launcher 設定、プロジェクト検出、スタートアッププロンプト生成
  • 状態管理 — state.local.json による install/update 状態の記録と復元
  • CLI 引数パース — --agent, --all, --dir, --source, --scope 等
HIGH Update F-01: ghcp / codex の update コマンドが install と同一
📄 src/setup/plugin-install.ts : 359–376
officialPluginCommands() は ghcp・codex について、action === 'update' を区別せず install と同じコマンドを生成する。Claude のみ claudePluginCommands() (L385–401) で update 専用パスがある。

ghcp/codex は marketplace add + plugin install がべき等の可能性があり、これが意図的な設計であれば問題ないが、明示的なドキュメントやコメントが不足。
💡 推奨: 意図的であればコードコメントで明記。意図的でなければ update 専用コマンドを追加。
HIGH Update F-02: Claude update コマンドの実在性が未検証
📄 src/setup/plugin-install.ts : 390–395
Claude 向けに claude plugin marketplace update と claude plugin update コマンドが定義されているが、これらの CLI コマンドが Claude Code CLI に実際に存在するか検証が必要。
💡 推奨: Claude Code CLI のドキュメント・実機で動作確認。存在しない場合は install ベースのアプローチに変更。
MEDIUM Update F-03: Update コマンドが state 欠損時に非インタラクティブで失敗
📄 bin/azure-functions-skills.js : 276–287
update は state ファイルからインストール済みエージェントを読み取るが、state が欠損/破損している場合、非インタラクティブモードではエラー終了する。detectAgents() へのフォールバックがない。
💡 推奨: state 欠損時に detectAgents() で自動検出し、ユーザーに確認。
LOW Chat F-04: Chat help で printHelp() が表示される
📄 bin/azure-functions-skills.js : 612–614
chat --help でコマンド固有のヘルプではなく全体ヘルプ (printHelp()) が表示される。他のコマンドは printCommandHelp(command) を呼んでいる。
💡 推奨: printCommandHelp('chat') に変更。

2. --dry-run 機能

✅ 正常動作を確認した領域

  • install/update: プラグインコマンドとワークスペースファイルの一覧を正しく表示 (L483–491)
  • plugin install/update: planPluginOperation() でコマンド一覧を返し、実行しない
  • workspace apply: ファイル一覧を plannedFiles として返す
  • doctor: --dry-run は N/A (実行前レポートの概念がない)
MEDIUM Dry-run F-05: --local --dry-run で表示される情報が不十分
📄 bin/azure-functions-skills.js : 438–442
install --local --dry-run では各エージェントに対して「workspace setup files」と表示するだけで、具体的にどのファイルが書き込まれるかが示されない。対して非ローカルモードの dry-run (L483–491) ではプラグインコマンドとファイルが一覧表示される。
💡 推奨: local モードでも buildTarget() の出力ファイル一覧をプレビュー表示。
LOW Dry-run F-06: Dry-run でも前提条件チェック内容が表示されない
📄 bin/azure-functions-skills.js, src/setup/index.ts
install --dry-run ではプラグイン操作とワークスペースファイルは表示されるが、前提条件チェック (Azure CLI, Core Tools 等) の計画は表示されない。
💡 推奨: 前提条件チェックの一覧も dry-run 出力に含める。

3. デッドコード

LOW Dead Code F-07: getPluginDir() の未使用パラメータ _target
📄 src/setup/plugin-install.ts : 84–86
getPluginDir(_target: BuildTargetName) はパラメータ _target を使用せず、常に同じパスを返す。アンダースコアプレフィックスで未使用を示しているが、将来の拡張性のために残しているならコメントで明記すべき。
💡 推奨: 使用予定がなければパラメータを削除。将来用ならコメント追記。
LOW Dead Code F-08: installPlugin / generateXxxSettings 系関数の利用範囲の限定
📄 src/setup/plugin-install.ts : 93, 107, 136, 148
generateVscodeSettings(), generateCodexMarketplaceEntry(), generateClaudeSettings(), installPlugin() はエクスポートされているが、メインの install/update フローでは使われず、レガシーの setup --as-plugin パス (L531–564) と chat --as-plugin (L640–661) でのみ使用。テストでもカバーされている。

デッドコードではないが、主要フローと別系統の実装が残っているため、メンテナンス負荷がある。
💡 推奨: パブリック API として維持するなら JSDoc に利用パターンを記載。不要なら内部関数に変更。
INFO Dead Code F-09: packageDependenciesCheck はスタブ状態
📄 src/doctor/checks.ts : 460–475 / docs/doctor-checks-best-practices-reference.md : 447
packageDependenciesCheck は ALL_CHECKS に登録されているがスタブ実装。ドキュメントでも「Stub; needs implementation」と記載されている。
💡 推奨: 実装するか、GA 前にスタブであることを明示。

5. Doctor コマンド

✅ 優れた設計・実装

  • Tier 1 (組み込み) / Tier 2 (AI) の明確な分離
  • サプライチェーンチェック (lifecycle scripts, unpinned deps, lockfile, tracked secrets, install-script deps, Python) が充実
  • PR コンテキスト検出で --deep の不正使用を防止 (isContributorPrContext())
  • --accept-deep-risk による明示的な同意要求
  • 4 種類の出力形式 (text, json, markdown, html)
  • Stacks API フォールバック戦略 (キャッシュ → フェッチ → 古いキャッシュ → ハードコード)
CRITICAL Doctor F-13: ドキュメントが未実装チェックを「実装済み」のように記載
📄 docs/doctor-checks-best-practices-reference.md : 124–136, 432–467
リファレンスドキュメントに SC-001 (ハードコードシークレット検出)、SC-002 (HTTP トリガー認証レベル)、CQ-001 (クライアント再利用パターン)、AS-006 (Application Insights) 等のチェックが記載されているが、実装は 20 チェック中これらは含まれていない。

実装状況テーブル (L436–451) は 13 チェック実装済みとしているが、実際の ALL_CHECKS (checks.ts L952–974) には 20 チェックが登録されている。
💡 推奨: 実装状況テーブルを最新化し、未実装チェックを「Planned」と明示。ユーザーに誤解を与えないよう区別を明確に。
HIGH Doctor F-14: Extension Bundle バージョン比較がメジャーバージョンのみ
📄 src/doctor/checks.ts : 153–159
extensionBundleCheck はバージョン範囲 [X.Y.Z, A.B.C) の比較でメジャーバージョンのみを比較。例えば [4.99.0, 5.0.0) と [4.0.0, 5.0.0) を区別できない。
💡 推奨: 完全なセマンティックバージョン比較を実装 (少なくとも minor まで)。
HIGH Doctor F-15: isContributorPrContext() が主要 3 CI のみ対応
📄 src/doctor/runner.ts : 318–325
PR コンテキスト検出が GitHub Actions, Azure DevOps, GitLab CI の 3 システムのみ。Jenkins, CircleCI, Travis CI 等のユーザーは検出されず、untrusted PR コードに対して --deep が実行される可能性がある。

ただし、コメント (L310–316) でヒューリスティクスとして 3 システムのみを明記しており、AZURE_FUNCTIONS_DOCTOR_TRUST_PR=1 というオーバーライドも用意されている。
💡 推奨: 汎用 CI 環境変数 (CI=true) のチェックを追加するか、ドキュメントでサポート対象 CI を明記。
MEDIUM Doctor F-16: readJsonFile() がエラー原因を区別しない
📄 src/doctor/checks.ts : 575–582 / src/doctor/context.ts : 8–14
JSON 読み取り関数がすべての例外を黙って捕捉し null を返す。ファイル不在と JSON パースエラーを区別できず、不正な host.json 等が黙って無視される。
💡 推奨: ファイル存在/パースエラーを区別し、パースエラー時は警告を出力。
MEDIUM Doctor F-17: gitignore パターンマッチが簡易的
📄 src/doctor/checks.ts : 767–775
isIgnored() のパターンマッチングは ** グロブ、! 否定パターン、サブディレクトリパターン等の gitignore フルセマンティクスに対応していない。
💡 推奨: 制限事項をドキュメント化するか、gitignore ライブラリを使用。
MEDIUM Doctor F-18: V4 関数検出のトリガータイプが限定的
📄 src/doctor/context.ts : 73
app\.(http|timer|storageBlob|storageQueue|serviceBus|eventHub|cosmosDB) のみ検出。app.eventGrid, app.durableClient, app.warmup 等の新しいトリガーが未対応。
💡 推奨: v4 モデルの全トリガータイプを網羅するか、汎用パターン app\.\w+\( を使用。

6. クロスプラットフォーム対応

✅ 評価: A (優秀)

クロスプラットフォーム設計は非常に高品質です：

  • path.join() による一貫したパス構築 — 全ソースファイルで実践
  • process.platform === 'win32' による適切な OS 分岐 (6 箇所以上)
  • os.tmpdir(), os.homedir() の正しい使用
  • Windows .cmd/.bat ラッパーの解決 (chat/index.ts L246–251)
  • パスセパレータ正規化 (replaceAll('\\', '/') — verify-plugin-payload.ts)
  • スクリプトの .sh/.ps1 デュアル提供 (health-status, inventory)
  • ハードコードされた絶対パス・ユーザー名・マシン名なし
HIGH Cross-Platform F-19: Codex SessionStart フックが bash 依存
📄 src/build/build-target.ts : 433–434
Codex の SessionStart フックが bash -c '...' を使用。Windows で bash がインストールされていない環境では動作しない。GHCP フックは Node.js ベースのインラインコマンドを使用しており、クロスプラットフォーム対応済み。
💡 推奨: GHCP と同様に node -e ベースのコマンドに変更するか、OS 検出でコマンドを分岐。
LOW Cross-Platform F-20: Doctor E2E テストスクリプトが Windows 専用
📄 scripts/doctor-e2e-setup.ps1, scripts/doctor-e2e-cleanup.ps1
E2E テスト用ヘルパースクリプトが PowerShell のみで、Linux/Mac の .sh 版がない。
💡 推奨: 対応する .sh バージョンを追加。

7. スキル品質レビュー

✅ Azure Functions 公式スキルとしての品質

  • 10 スキルが体系的に設計されており、ワークフロー全体を網羅 (setup → create → deploy → diagnose → review)
  • MCP ファースト + フォールバックの一貫した設計パターン
  • 承認ゲート付きの外部アーティファクト作成 (feedback スキル)
  • シークレット墨消しルールの明記
  • スキル名が azure-functions-{verb} パターンで統一、発見しやすい
  • バンドルスクリプトによるプラットフォーム別実行 (.sh / .ps1)
CRITICAL Skills F-21: HTTP トリガー検証ガイダンスがスキル間で矛盾
📄 templates/skills/azure-functions-deploy/SKILL.md : L106
📄 templates/skills/azure-functions-create/SKILL.md : L123
azure-functions-deploy は「curl -I (HEAD) は偽陰性になるため使用しないこと」と明記。 しかし azure-functions-create は HTTP 検証方法について GET/HEAD を明示しておらず、矛盾するガイダンスにつながる可能性がある。
💡 推奨: 全スキルで「GET リクエストを使用、HEAD は使用しない」を統一的に記載。
HIGH Skills F-22: azure-functions-create のタイマートリガー検証で本番事故リスク
📄 templates/skills/azure-functions-create/SKILL.md : 128–133
タイマートリガーの検証時に「開発用に短いスケジュールに一時変更」を推奨しているが、元のスケジュールに戻すことの重要性が十分に強調されていない。本番でショートスケジュールが残る事故につながる可能性がある。
💡 推奨: ⚠️ 警告付きで「必ず元のスケジュールに復元すること」を明記し、検証完了チェックリストに含める。
HIGH Skills F-23: ローカルエミュレータ参照が不完全
📄 templates/skills/azure-functions-common/references/local-emulators.md : 31–33
MySQL, RabbitMQ, Dapr のエミュレータについて「No Azure Functions-specific emulator page found」と記載。公式スキルとしてはギャップに見える。

ただし Azure Storage Emulator (引退済み) → Azurite への移行は正しく記載されている。
💡 推奨: コミュニティ推奨の代替手段を記載するか、「Azure Functions 固有の公式ドキュメントが存在しないサービス」と明確にラベル付け。
MEDIUM Skills F-24: Extension Bundle マニフェストリンクが main ブランチ参照
📄 templates/skills/azure-functions-common/references/extensions/extension-bundles.md : L17
Extension Bundle マニフェストへのリンクが GitHub の main ブランチを指している。スキーマ変更があった場合に参照が壊れるリスク。
💡 推奨: バージョン付きタグまたは安定版リンクに変更。
MEDIUM Skills F-25: プログラミングモデル v1 の非推奨が明示されていない
📄 templates/skills/azure-functions-common/references/languages/
Python v1, Node.js v1/v2/v3 のプログラミングモデルが「古い」と示唆されているが、「非推奨」または「サポート終了」とは明記されていない。ユーザーがまだ使えると誤解する可能性がある。
💡 推奨: 各言語リファレンスに明確なサポートマトリクス（現行バージョン / 非推奨バージョン）を追加。
MEDIUM Skills F-26: health-status / inventory の Bash スクリプトが Python 必須だが記載なし
📄 templates/skills/azure-functions-health-status/scripts/get-functionapp-health-status.sh
📄 templates/skills/azure-functions-inventory/scripts/get-functionapp-inventory.sh
Bash スクリプトが内部で Python を使用しているが、SKILL.md にその前提条件が記載されていない。
💡 推奨: SKILL.md に Python 依存を明記するか、pure-Bash 実装に変更。
LOW Skills F-27: azure-functions-common のスキル名が内部用と分かりにくい
📄 templates/skills/azure-functions-common/SKILL.md
「Not intended as a standalone user-facing workflow」と記載されているが、スキル名からそれが読み取れない。ユーザーが直接呼び出す可能性がある。
💡 推奨: azure-functions-reference-library 等の名前を検討するか、スキル説明に「他のスキルから参照される内部リファレンス」と強調。
LOW Skills F-28: azure-functions-doctor と azure-functions-diagnostics の名前が紛らわしい
📄 templates/skills/azure-functions-doctor/, templates/skills/azure-functions-diagnostics/
doctor = デプロイ前の静的分析、diagnostics = ランタイムトラブルシューティング。用途は異なるが、名前が類似しており混同されやすい。
💡 推奨: スキル説明の冒頭で違いを明確に記載 (例: "Pre-deploy static analysis" vs "Runtime troubleshooting")。

8. ワークスペース構成の整合性

✅ 確認済みの整合性

  • 3 エージェント用ルーティングファイル (ghcp.md, claude.md, codex.md) が同一テンプレートパターンで統一
  • MCP サーバー設定 (servers.yaml) が正しく @azure/mcp@latest を参照
  • プラグイン marketplace.json (.plugin/, .claude-plugin/) が同一コンテンツ
  • スタートアッププロンプト、Welcome フックが適切にテンプレート変数を使用
  • 孤立した未使用コンポーネントなし
MEDIUM Workspace F-29: AGENTS.md の内容がリポジトリ開発標準とユーザープロジェクト標準で曖昧
📄 templates/agents/AGENTS.md
このファイルはユーザーのワークスペースにインストールされ、コーディングエージェントに開発標準を指示する。内容は「TDD」「Conventional Commits」「PR ガイダンス」等の汎用開発標準と、Azure Functions 固有の標準が混在。

Azure Functions 固有の部分 (L34–38) は適切だが、「Small Commits」「Conventional Commits」等はユーザーのプロジェクトに必ずしも適合しない可能性がある。
💡 推奨: Azure Functions 固有の標準に特化し、汎用開発標準は任意適用のセクションとして分離するか、ユーザーがカスタマイズしやすいよう明記。
MEDIUM Workspace F-30: Azure Skills インストールガイダンスがスキル間で分散
📄 templates/skills/azure-functions-setup/SKILL.md : L40–45
📄 templates/skills/azure-functions-deploy/SKILL.md : L80–88
📄 templates/hooks/welcome-setup.md : L30–43
Azure Skills (MCP サーバー) のインストール方法がスキルやフック間で分散しており、一元的なリファレンスがない。エントリポイントによって異なるインストール手順が表示される可能性がある。
💡 推奨: 共通リファレンスファイルに一元化し、各スキルからリンク。
LOW Workspace F-31: marketplace.json の同期メカニズム未整備
📄 .plugin/marketplace.json, .claude-plugin/marketplace.json
両ファイルは同一内容だが、バージョンバンプ時に片方だけ更新される可能性がある。同期を保証するビルドチェックがない。
💡 推奨: CI に両ファイルの同一性チェックを追加。

9. ドキュメントレビュー

✅ ドキュメントの良い点

  • CLI リファレンスが全コマンド・オプションを網羅
  • セキュリティモデルが明確に文書化 (SECURITY.md, doctor-guide.md)
  • 開発者向けドキュメント (development.md) が充実
  • usage-scenarios.md で実用的なシナリオが提示
  • 日本語ドキュメント (doctor-checks-best-practices-reference.ja.md) の提供
CRITICAL Docs F-32: 内部ドキュメントの公開前整理
📄 docs/research/bug-bash-feedback-summary-20260519.md
📄 docs/research/bug-bash-chat-test-steps-20260519.md
📄 docs/internal/
📄 docs/prd-docs/
